An argument for TDD

A post about the Law of Demeter

September 20, 2017 Nathaniel Saul

3 minute read

In this post, We propose a new software development methodology dubbed debug-driven development. Building upon work out of context and ignoring basic principles of time management, our proposed method cuts the average development* time of a project in half.

By using test driven development, we can redefine the develop-debug process to completely eliminate the development step. Instead of developing a feature, we can fix a broken test and in doing so the development portion of the develop-debug process disappears.

Background

Debugging and fixing defects amounts to half the cost of software development 1. Half of the time you spend writing code, you’re not actually writing code, your fixing code. The process of developing is a constant ping-pong of develop, debug, develop, debug. While developers have devoted immense effort into decreasing both the development cost and debug cost of a project.

Test driven development is the process of writing your tests first. Before developing any code whatsoever, you first create a test that once passing, will tell you the code you created works how you originally wanted. This methodology has been heralded by many to improve code quality, reliability, maintainability, and development time. At first this seems counter-intuitive as you end up writing so much code for tests that are trivial and don’t need testing anyways.

Writing tests first improve code on a number of different fronts. None of these benefits are only realized through TDD. The real benefit comes not exactly from writing the tests, but from always thinking about how would i test this. It requires you to think first and foremost about what the program should do, not how it gets it done. This generally yields more intuitive interfaces that are transparent. Having all the tests in place once you’re done of course increases maintainability and reliability.

Related work

Many researchers have explored the notion of bug, many finding that often bugs can be reinterpreted as features. While some industries adopt the mantra the customer is always right, researchers have found that development time can be greatly decreased by ignoring customer requests and developing whichever features they wish. This area has been both intriguing academically and has found stable footing within industry [cite shitty companies]

Contributions

By adopting test driven development, we can reframe a desired feature to actually be a bug in the code. The development time of implementing the feature then gets transformed into debugging time. Since debugging amounts to half of the time spent developing software and we are only debugging code, the entire timeline for the project instantly is cut in half.

1 Reworking defects in requirements, design, and code consumes 40-50% of the total cost of software development. (Boehm, Barry W. 1987. “Improving Software Productivity.” IEEE Computer, September: 43-57.; Jones, Capers, ed. 1986. Tutorial: Programming Productivity: Issues for the Eighties, 2nd ed.
